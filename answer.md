# Обеспечить разработку
Для организации процесса разработки предлагается использовать Gitlab.

Схема:
Каждый микросервис размещается в отдельном Git-репозитории внутри группы GitLab:
* company/api-gateway;
* company/auth-service;
* company/user-service;
* company/payment-service;
* company/notification-service.

В каждом репозитории хранится исходный код сервиса, Dockerfile, тесты и файл .gitlab-ci.yml, описывающий pipeline сборки, тестирования, публикации Docker-образа и деплоя.

Общая схема работы:

* Разработчик делает push или merge request в GitLab.
* GitLab автоматически запускает pipeline.
* GitLab Runner выполняет этапы сборки, тестирования и публикации артефактов.
*Docker-образ сервиса публикуется в GitLab Container Registry.
* На этапе CD образ разворачивается в нужную среду: dev, stage или prod.
* Секреты, токены и пароли хранятся в GitLab CI/CD Variables или во внешнем хранилище секретов, например HashiCorp Vault.

| Требование                                                               | Реализация                                                                                |
| ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------- |
| Облачная система                                                         | Используется GitLab.com                                                                   |
| Система контроля версий Git                                              | GitLab предоставляет Git-репозитории                                                      |
| Репозиторий на каждый сервис                                             | Для каждого микросервиса создаётся отдельный репозиторий                                  |
| Запуск сборки по событию из системы контроля версий                      | Pipeline запускается по push, merge request, tag                                          |
| Запуск сборки по кнопке с указанием параметров                           | Manual pipeline запускается из UI GitLab с передачей переменных                           |
| Возможность привязать настройки к каждой сборке                          | Используются переменные pipeline, environment variables, rules, stages                    |
| Возможность создания шаблонов для различных конфигураций сборок          | Используются `include`, `extends`, hidden jobs, общие CI-шаблоны                          |
| Возможность безопасного хранения секретных данных: пароли, ключи доступа | Используются GitLab CI/CD Variables: masked, protected, hidden; при необходимости — Vault |

GitLab удобен для микросервисной архитектуры, потому что в одном продукте объединены Git-репозитории, CI/CD, Container Registry, управление секретами, шаблоны pipeline, ручные запуски, self-hosted runners и разграничение прав доступа. Для крупной компании это снижает количество отдельных систем и упрощает сопровождение процесса разработки.


# Логи
Для сбора и анализа логов предлагается использовать связку:

Vector + Apache Kafka + Elasticsearch + Kibana.



Схема:

* Каждый микросервис пишет логи в stdout и stderr.
* На каждом сервере или Kubernetes-ноде установлен Vector Agent
* Vector собирает контейнерные логи из Docker или Kubernetes.
* Vector добавляет служебные поля: имя сервиса, имя контейнера, хост, окружение, timestamp.
* Vector отправляет события в Kafka.
* Vector Aggregator или Logstash читает данные из Kafka и отправляет их в Elasticsearch.
* Kibana используется как пользовательский интерфейс для поиска, фильтрации, построения сохранённых поисков и выдачи ссылок разработчикам.

Kafka нужна как центральный отказоустойчивый буфер. Если Elasticsearch временно недоступен, логи не теряются, а остаются в Kafka в рамках настроенного retention. После восстановления Elasticsearch обработчик продолжит чтение сообщений.

| Требование                                                                                                               | Реализация                                                                                                                  |
| ------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------- |
| Сбор логов в центральное хранилище со всех хостов, обслуживающих систему                                                 | Elasticsearch используется как центральное хранилище логов. Vector Agent устанавливается на каждый хост или Kubernetes-ноду |
| Минимальные требования к приложениям, сбор логов из stdout                                                               | Приложения просто пишут логи в `stdout/stderr`                                                                              |
| Гарантированная доставка логов до центрального хранилища                                                                 | Используются Vector disk buffer, Kafka replication и повторная доставка в Elasticsearch                                     |
| Обеспечение поиска и фильтрации по записям логов                                                                         | Используются Kibana Discover и Elasticsearch Query DSL                                                                      |
| Обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов | Используется Kibana с ролевой моделью доступа                                                                               |
| Возможность дать ссылку на сохранённый поиск по записям логов                                                            | В Kibana можно сохранить Discover search и передать ссылку                                                                  |
Vector лёгкий, быстрый и удобный для сбора логов из контейнеров. Kafka повышает надёжность доставки и позволяет переживать временную недоступность Elasticsearch. Elasticsearch хорошо подходит для хранения и полнотекстового поиска по логам, а Kibana предоставляет удобный веб-интерфейс для разработчиков и эксплуатации.

# Мониторинг

Для мониторинга предлагается использовать:

Prometheus + Node Exporter + cAdvisor + Grafana + Alertmanager.

Схема:
* На каждом хосте устанавливается Node Exporter.
* Для контейнеров используется cAdvisor.
* Каждый микросервис предоставляет endpoint /metrics.
* Prometheus периодически собирает метрики с exporters и сервисов.
* Grafana подключается к Prometheus как источник данных.
* В Grafana создаются dashboards для хостов, сервисов и бизнес-метрик.
* Alertmanager используется для отправки уведомлений при авариях.

Метрики хостов собираются через Node Exporter.
Предлагаю собирать - CPU usage, RAM usage, disk usage, disk I/O, network traffic, load average, load average.
Метрики контейнеров - CPU, RAM по контейнеру, network rx/tx, filesystem usage, количество рестартов, состояние контейнера.

| Требование                                                                                                | Реализация                                           |
| --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| Сбор метрик со всех хостов, обслуживающих систему                                                         | Node Exporter устанавливается на каждом хосте        |
| Сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network                                             | Используется Node Exporter                           |
| Сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network                             | Используется cAdvisor или Kubernetes metrics         |
| Сбор метрик, специфичных для каждого сервиса                                                              | Каждый микросервис предоставляет `/metrics` endpoint |
| Пользовательский интерфейс с возможностью делать запросы и агрегировать информацию                        | Используются Prometheus UI и Grafana Explore         |
| Пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы | Используются Grafana dashboards                      |


Prometheus является стандартным решением для сбора метрик в микросервисной архитектуре. Он хорошо работает с pull-моделью, service discovery, exporters и собственными метриками приложений. Grafana предоставляет удобный интерфейс для построения dashboards, анализа и визуализации состояния системы. Node Exporter закрывает уровень хостов, cAdvisor — уровень контейнеров, а /metrics — уровень бизнес-логики конкретных сервисов.
